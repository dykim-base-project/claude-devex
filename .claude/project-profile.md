# Project Profile

프로젝트별 특수성을 정의하여 `/spec`과 `/implement` 스킬의 동작을 조정합니다.

## 산출물 유형

| 구분 | 내용 |
|------|------|
| 주요 산출물 | 마크다운 (스킬 파일, 가이드), 쉘 스크립트 |
| 빌드 도구 | 없음 |
| 테스트 방법 | `setup.sh`를 빈 디렉토리에서 실행하여 검증 |

## /spec 컨텍스트

`/spec` 호출 시 아래 관점으로 명세를 작성합니다.

- **설계 대상**: 스킬 파일(SKILL.md), 쉘 스크립트, 가이드 문서
- **명세 항목**:
  - 변경 전/후 비교 (기존 스킬 내용 vs 변경안)
  - 적용 사례 레포 영향 분석 (keycloak-practice 등)
  - 업데이트 안전 영역 확인 (setup.sh로 전파 시 보존/덮어쓰기 대상)
- **다이어그램**: 워크플로우 변경 시 Mermaid로 표현

## /implement 컨텍스트

`/implement` 호출 시 아래 관점으로 구현합니다.

- **구현 대상**: SKILL.md 파일 편집, setup.sh 업데이트, 가이드 문서 수정
- **구현 순서**:
  1. 스킬 파일 변경
  2. setup.sh 반영 (필요 시)
  3. README.md / .claude/README.md 동기화
- **검증 기준**:
  - 스킬 파일 6개 존재 확인 (github-issue, spec, implement, commit, github-pr, cycle)
  - `setup.sh`를 빈 디렉토리에서 실행하여 정상 설치 확인
  - README.md Mermaid 다이어그램 렌더링 확인
  - 범용성 유지: 특정 프로젝트에 종속되는 내용 없음

## 제약사항

- 스킬 파일은 이 레포의 **제품**이므로 범용성을 반드시 유지
- 스킬 변경 시 적용 사례 레포에도 동기화 필요
- `/spec` 단계에서 스킬 변경 명세를 먼저 작성
